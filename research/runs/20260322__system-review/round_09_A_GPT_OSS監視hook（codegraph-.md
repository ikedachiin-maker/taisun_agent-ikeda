## 総合評価: F

### 1. 問題の深刻度判定
今回の提案書は、**テーマが「codegraph-oss-monitor.js の GitHub API 呼び出しとレート制限リスク」なのに、肝心の論点をほとんど評価できていません。**  
その意味で、深刻度判定そのものがかなり甘く、論点逸脱もあります。

以下、主な問題点です。

1. **GitHub APIレート制限リスクの定量評価がない**  
   - **深刻度: Critical**  
   - SessionStartで毎回 `codegraph-oss-monitor.js` が動く構成なのに、  
     - 認証あり/なしの呼び出し上限
     - 1セッションあたりのAPI消費量
     - 同時セッション数を踏まえた枯渇見積もり
     - 二次レート制限や abuse detection  
     が一切分析されていません。テーマに対する本丸の欠落です。

2. **SessionStart同期フックとしての可用性リスクを軽視している**  
   - **深刻度: High**  
   - 10秒timeoutのフックが外部API依存なのは危険です。GitHub API遅延・障害・DNS不調・TLS問題で、起動体験やセッション開始処理全体に悪影響が出ます。  
   - 提案書は「hookが多い」「12秒待つ価値」には触れているが、**SessionStartで外部依存を同期実行する設計不備**を問題化していません。

3. **キャッシュ戦略・条件付きリクエストの有無を確認していない**  
   - **深刻度: High**  
   - ETag/If-None-Match、Last-Modified、TTLキャッシュ、ローカル永続キャッシュの有無はレート制限回避の基本です。  
   - これを見ずに健全性監視を評価するのは不十分です。

4. **失敗時のフォールバック設計を見ていない**  
   - **深刻度: High**  
   - GitHub APIが429/403/5xxのときに、  
     - リトライするのか
     - fail-openか fail-closedか
     - バックオフするのか
     - 監視結果を stale cache で代替するのか  
     が不明です。  
   - OSS監視は補助機能のはずで、失敗で本処理を阻害するなら設計ミスです。

5. **認証方式とトークン運用リスクを見落としている**  
   - **深刻度: Medium**  
   - GitHub APIの未認証呼び出しは上限が低く、共有IP環境では即死しやすいです。  
   - PAT/GitHub App利用なら権限最小化、ローテーション、漏えい時影響も見るべきですが、提案書は未評価です。

6. **API呼び出し対象と頻度の妥当性評価がない**  
   - **深刻度: Medium**  
   - OSS健全性監視で何を見るのか不明です。stars/forks/open issues/latest release/security advisory等を毎回取りに行く必要があるのか、SessionStartごとに実行すべきか、差分確認で足りるのかの設計評価がありません。  
   - 「必要なデータだけに絞る」観点が欠けています。

7. **観測性不足を指摘していない**  
   - **深刻度: Medium**  
   - レート制限系は、`x-ratelimit-remaining`, `x-ratelimit-reset`, ステータスコード、応答時間、キャッシュヒット率を記録しないと運用不能です。  
   - 提案書は計測一般論はあるが、**このhookの運用指標**に落ちていません。

8. **セキュリティ評価が表層的**  
   - **深刻度: Medium**  
   - 「公式リリースからダウンロード」だけでは不十分です。  
   - バイナリ署名/チェックサム検証、SBOM、脆弱性追跡、実行権限、更新経路、ネットワーク到達先制限まで見るべきです。  
   - 特に外部APIへ出るhookとバイナリ導入が絡むので、サプライチェーン観点が弱いです。

9. **M3の評価が甘い**  
   - **深刻度: Medium**  
   - `pre-compact-save.js` を「可能性」に留めていますが、Bashごとに最大5秒は実運用でかなり危険です。  
   - 一方で、今回のテーマでより危険な `codegraph-oss-monitor.js` の方を相対的に重く扱っていないのは優先度判断ミスです。

10. **構成表の整合性チェックが甘い**  
    - **深刻度: Low**  
    - MCPサーバー数が「16」と書かれている一方で列挙数が一致していないように見え、無効化済み一覧との関係も曖昧です。  
    - こうした基礎データの不整合はレビュー全体の信頼性を下げます。

### 2. 修正優先順位
優先順位は以下です。

1. **codegraph-oss-monitor.js の GitHub API呼び出し実態の可視化**  
   - 呼び出し回数/セッション、エンドポイント、認証有無、失敗率、平均応答時間、レートヘッダを記録
2. **SessionStart同期実行の停止または非同期化**  
   - まず fail-open にし、セッション開始をブロックしない設計へ
3. **キャッシュ導入**  
   - TTL + ETag/If-None-Match + stale-while-revalidate
4. **429/403/5xx時のバックオフとフォールバック実装**
5. **認証方式の整理**  
   - PATよりGitHub App優先、最小権限化
6. **必要データの最小化**
7. **観測性追加**
8. **その他hookの性能最適化**

即座に修正すべきものは、**外部API依存のSessionStartブロッキング設計**です。

### 3. 未使用hookの整理方針
未使用hookは無条件削除ではなく、以下の3分類で処理すべきです。

- **削除**: 90日以上未使用、所有者不明、テストなし、再利用予定なし
- **隔離保管**: 将来候補だが未成熟なもの。`hooks/experimental/` へ移動
- **保持**: 明確なロードマップ・オーナー・有効化条件があるもの

現状のままルートに20+放置は悪手です。探索コスト、誤有効化、保守負債を増やします。

### 4. パフォーマンス懸念
あります。  
ただし「17本だから危険」ではなく、**イベントごとの同期実行合計時間**が問題です。

特に危険なのは以下です。
- SessionStart: `workflow-sessionstart-injector.js` + `codegraph-oss-monitor.js`
- PreToolUse: Write/Edit/Bashに複数ガードが多重起動
- PostToolUse: Write/Editで `codegraph-auto-index.js` が最大12秒

今回の提案書は全hookの合計時間計測を要求している点はよいですが、**外部I/O依存hookとローカルhookを分けて評価すべき**です。外部I/O依存は分散が大きく、平均値だけでは危険を隠します。

### 5. CodeGraph導入の影響
設計に問題があります。

- `codegraph-auto-index.js` のWrite/Editごとの再索引は過剰の可能性
- `codegraph-oss-monitor.js` をSessionStartで毎回走らせるのは責務配置が悪い
- `codegraph-roi-meter.js` はRead/Grep/Glob後で軽量ならよいが、計測自体がノイズになる恐れ

特に `OSS監視` はCodeGraph本体のインデックス更新責務とは別です。  
**起動時の必須処理に置く理由が弱い**。日次・手動・バックグラウンドで十分な可能性が高いです。

### 6. セキュリティ
現評価は甘いです。

- Goバイナリは**チェックサム固定、署名確認、配布元固定**が必要
- hookのNode実行は安全ではありません。`child_process`, `exec`, `fetch`, FSアクセス範囲を確認すべきです
- MCP設定は、無効化済みも含めて過大です。設定漂流の兆候があります
- GitHub APIトークン利用があるなら、権限境界・保存先・マスキング・監査ログが必要

「shell injection対策は各hookで個別実装」はむしろ危険です。**共通安全ライブラリ化されていない**という意味だからです。

### 7. スケーラビリティ
劣化は起きます。  
特に問題なのは件数そのものではなく、**組み合わせ爆発による認知負荷・設定複雑性・起動時の分散遅延**です。

- 700+スキル: 発見性低下、重複、陳腐化
- 17 hook: 競合、順序依存、デバッグ困難
- 16+ MCP: 接続失敗時の影響範囲拡大、設定ドリフト

提案書は規模の大きさを示すだけで、**運用上の上限設計**を示していません。  
例:
- hook予算: イベント別の総実行時間上限
- MCP予算: 同時有効数の上限
- skill寿命: 未使用スキルの棚卸し基準

### 8. 最も弱い設計箇所TOP 3
1. **SessionStartに外部API依存hookを置く設計**  
   - 起動可用性とレート制限耐性が低い
2. **hook群の責務分散と順序依存の不透明さ**  
   - ガード、監視、索引、最適化が散在し、失敗影響が読みにくい
3. **未使用資産と有効資産が混在する構成管理**  
   - hook/MCP/skillの棚卸し不全で、保守性が下がっている
