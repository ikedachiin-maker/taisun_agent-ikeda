### 1. 問題の深刻度判定

前回監査の深刻度判定は全体として甘いです。特に以下は再評価が必要です。

#### 指摘1: Go製136MBバイナリを「公式リリースからダウンロード」で許容している
- **深刻度: Critical**
- 問題は「公式かどうか」ではありません。**再現可能ビルド、署名検証、SHA256固定、SBOM、脆弱性スキャン、配布元信頼性、更新ポリシー**が欠落しています。
- ソースビルドでない以上、**サプライチェーン攻撃面**が最大の懸念です。
- しかもAIエージェント環境に組み込み、コードベース全体を読ませる設計であり、侵害時の影響範囲が大きすぎます。

#### 指摘2: codegraph-auto-index.js を Write/Edit 後に自動実行
- **深刻度: High**
- 「便利機能」として扱っていますが、これは**書き込みイベントに連動する高頻度・高コスト・状態更新処理**です。
- timeout 12s は異常に長く、**開発体験悪化、競合、インデックス不整合、編集ループ、部分書き込み状態の誤取り込み**を招きます。
- さらに、Write/Editのたびに外部バイナリまたはDB更新に触れる設計は、**障害点をホットパスに混入**させています。

#### 指摘3: hookの権限が「node実行、shell injection対策は各hookで個別実装」
- **深刻度: Critical**
- これは対策ではなく**統制不在の宣言**です。
- セキュリティ上、入力検証やサニタイズを各hook作者に委ねるのは最悪です。  
  **共通実行ラッパー、許可コマンド制、引数エスケープ標準化、権限分離、監査ログ**が必要です。
- 個別実装は品質ばらつきとレビュー漏れを必ず生みます。

#### 指摘4: pre-compact-save.js の Bashフックを Medium 扱い
- **深刻度: High**
- Bash実行ごとに最大5秒遅延し得る設計は、単なる性能問題ではなく、**操作阻害と予測不能な副作用**を生むため重大です。
- Bashは最も頻繁で影響の大きいトリガーの一つです。そこでI/O系保存処理を入れるのは設計不適切です。

#### 指摘5: codegraph-oss-monitor.js を SessionStart で10s許容
- **深刻度: Medium**
- OSS健全性監視をセッション開始時に同期的に走らせるのは悪手です。
- セッション開始はユーザー体感に直結するため、**ブロッキング監視**を置くべきではありません。
- ただし致命傷ではなく、非同期化・別ジョブ化で解消可能です。

#### 指摘6: supertest / Prismaの扱いを High に置いている
- **深刻度: Medium**
- これは改善対象ではあるものの、今のシステム全体リスクと比較すると優先度を上げすぎです。
- 依存分類ミスやCLI不足は**可用性/整備性の問題**であって、CodeGraph導入のセキュリティ/設計リスクより下です。

#### 指摘7: 17本hook + 20+未使用hookを放置
- **深刻度: High**
- 未使用hookが残ること自体より、**何が現役で何が死蔵か、誰が所有し、どの品質基準で維持されているか不明**な点が問題です。
- これは設定ドリフト・レビュー漏れ・誤有効化・監査不能に直結します。

#### 指摘8: MCPサーバー数の記載が不整合
- **深刻度: Medium**
- 「16サーバー」と書きながら列挙数が一致していないように見えます。  
- この種の台帳不整合は、**無効化済み/有効状態/実際の接続可能状態の監査不備**を示唆します。
- セキュリティレビューの前提データが揺らいでいる時点で危険です。

#### 指摘9: CodeGraphのDB 58MB SQLiteをローカル運用する際の保護設計がない
- **深刻度: High**
- インデックスDBには、コード構造・依存関係・設計意図に近い情報が凝縮されます。
- これは**ソース本体に準ずる機密資産**であり、保存場所、暗号化要否、アクセス権、破棄ポリシー、Git混入防止が必要です。
- 提案書はその扱いが極めて雑です。

#### 指摘10: 成功基準がセキュリティではなく体感中心
- **深刻度: Medium**
- 「p95 < 200ms」「ビルドサイズ10%削減」はあっても、  
  **署名検証100%、SBOM生成、hook失敗時fail-open/fail-close方針、権限境界テスト**がない。
- つまり成功基準が本質からずれています。

### 3. 未使用hookの整理方針

結論として、**原則削除、例外的に隔離保管**です。

保持すべきという意見はありますが、その場合でも本番同等ツリーに置くべきではありません。  
未使用hookが残るデメリットは以下です。

- 誤有効化リスク
- セキュリティレビュー対象の肥大化
- 実装品質の劣化放置
- 所有者不明コードの増殖
- 「存在しているだけで安心」の擬似統制

#### 推奨方針
- settings.json未登録のhookは **active treeから除去**
- 残す場合は `.claude/hooks/archive/` などへ退避
- 各hookに以下メタデータを必須化  
  - owner  
  - purpose  
  - trigger  
  - input schema  
  - side effects  
  - timeout  
  - last verified  
  - security review status
- 30日未使用で自動アーカイブ、90日で削除候補

今のような「未使用だが置いてある」は最悪です。

### 4. パフォーマンス懸念

**問題です。かなり。**

「17本あるが毎回全部走るわけではない」という反論はあり得ますが、重要なのは**ホットパスに複数hookが重複していること**です。

#### 特に問題な箇所
- **Write/Edit**  
  unified-guard, deviation-approval-guard, agent-enforcement-guard, auto-adr, definition-lint-gate, agent-trace-capture, compact-optimizer, output-verifier, codegraph-auto-index  
  → 実質的に非常に重い
- **Bash**  
  unified-guard, deviation-approval-guard, pre-compact-save  
  → 1コマンドごとに足回りが重い
- **SessionStart**  
  2本で5s/10s  
  → 初動遅延を誘発

#### 設計上の問題
- timeout値が性能目標ではなく「我慢限界値」になっている
- 同一イベントに複数hookがぶら下がり、責務分割より**責務分散**になっている
- PostToolUse `*` 系が広すぎる
- メトリクス収集計画はあるが、**導入前のベースライン欠如**

したがって、「17本が毎回動作すること」は誇張でも、**頻出イベントの合成オーバーヘッドは十分に実害化する**と判断します。

### 5. CodeGraph導入の影響

設計としてかなり弱いです。

#### 問題点
1. **重いバイナリをローカルに直接導入**
   - 配布・更新・検証・脆弱性管理が弱い
2. **3 hook追加でイベント面積を拡大**
   - ROI計測、auto-index、OSS monitor がそれぞれ別軸で動作
3. **自動インデックス更新の責務がhot pathにある**
   - バックグラウンドジョブにすべき
4. **CodeGraph導入の便益に対する失敗時設計がない**
   - 失敗時に無効化？再試行？破損DB復旧？不明
5. **CodeGraph DBをセキュアアセットとして扱っていない**
6. **ROI計測hookの存在が本質機能に寄与しない**
   - 監視用hookを本番運用に常駐させるべきか再検討が必要

#### 総評
CodeGraph自体の発想は否定しませんが、**導入方法が雑**です。  
「便利だから入れる」レベルの変更ではなく、**エージェントの信頼境界を変える変更**として扱うべきでした。

### 6. セキュリティ

現状評価は厳しく言って**不十分**です。

#### 主な懸念
- **未署名/未固定ハッシュのバイナリ**
- **個別hook実装にサニタイズを委任**
- **MCPサーバー群の最小権限原則が見えない**
- **CodeGraph DBの機密性保護なし**
- **hook失敗時の動作方針不明**
- **外部通信有無、テレメトリ有無、アップデート方式不明**
- **Node hook + Go binary + MCPの複合実行面**

#### 追加で確認すべき項目
- バイナリのchecksum固定はあるか
- GitHub Release以外のミラー混入を防げるか
- バイナリに自己更新機構がないか
- SQLite DBにPII/秘匿実装情報が含まれないか
- hooksから任意path読み取りが可能か
- MCP設定で不要サーバーが本当に無効か
- ログにコード断片や秘密情報が漏れないか

「ハードコードAPIキーなし」は、セキュリティレビューとしては初歩項目に過ぎません。

### 7. スケーラビリティ

#### 特に悪い兆候

- 未使用hookが多数残存
- 有効/無効/存在の3状態が混在
- 計測より先に機能追加している

規模の問題というより、**運用モデルがスケールしていない**です。
