### 1. 問題の深刻度判定
全体として甘いです。特に「hook 17本のオーバーヘッド」と「pre-compact-save.jsの5秒タイムアウト」の評価が雑です。

- **H2: prisma CLI不在**  
  **HighではなくCritical寄り**です。`@prisma/client`だけあって`prisma` CLIがない構成は、新規環境・CI・Docker buildで即死し得ます。しかも「失敗する可能性」ではなく、`postinstall`/`generate`運用次第でかなり高確率で壊れます。

- **H1: supertest本番混入**  
  Highは概ね妥当。ただし理由が弱いです。「ビルドサイズ増大」だけでは弱く、**SBOM/脆弱性監査/ライセンス監査/本番最小化原則違反**まで触れるべきでした。

- **M3: pre-compact-save.js**  
  **MediumではなくHigh候補**です。「Bash実行ごとに timeout 5秒」は、頻度依存ですが設計としてかなり危険です。  
  しかも記述が不正確で、**“最大5秒遅延”と“常時5秒遅延”を混同**しています。timeoutは上限であり、実測なしに結論を出している点が雑。

- **CodeGraph関連が未評価に近い**  
  `codegraph-auto-index.js` の **Write/Edit後 12s timeout** は、実運用の体感遅延・競合・再入・インデックス破損耐性の観点で十分にHigh検討対象です。レビューはここを軽視しすぎです。

- **セキュリティ評価が甘い**  
  「公式リリースからダウンロード」だけでGoバイナリ導入を良しとしているが、**署名検証・チェックサム固定・SBOM・更新経路の信頼性**が未確認。  
  これは少なくとも **Medium** で監査事項です。

### 3. 未使用hookの整理方針
提案書はここも浅いです。**未使用hookを一律削除/保持と判断するのは危険**です。

厳密には3分類すべきです。

- **即削除**
  - 所有者不明
  - テストなし
  - settings未登録
  - 3か月以上参照なし
  - セキュリティ敏感な処理を含む

- **アーカイブ保持**
  - 将来復帰の可能性あり
  - ただし `archive/hooks-disabled/` 等へ退避
  - READMEに「無効理由」「復帰条件」記載

- **再評価対象**
  - 現行課題に関係するもの
  - ただし有効化前にSLO・ベンチ・失敗時挙動を確認

問題は、**未使用hookが20+存在すること自体が運用負債**だという点です。  
「存在するが未登録」は安全ではありません。レビュー対象者が誤って再有効化し、設計判断の履歴も失われます。

### 4. パフォーマンス懸念
ここが今回のレビューで最も弱い部分です。

**結論：17本という本数だけでは判断不能。だが現状設計は十分に危険。**  
特に `pre-compact-save.js 5s` を許容とみなすのはかなり危ないです。

重要なのは本数ではなく以下です。

- **どのイベントで発火するか**
- **同期ブロッキングか**
- **高頻度イベントにぶら下がっているか**
- **I/Oを伴うか**
- **タイムアウト上限ではなく実測がどうか**
- **複数hookの直列実行か並列実行か**
- **失敗時フェイルオープン/フェイルクローズか**

特に危険なのは以下。

- `PreToolUse` に複数本
- `PostToolUse` に `*` matcher が存在
- `Write/Edit/Bash` の高頻度イベントに重い処理
- `codegraph-auto-index.js` の 12s timeout
- `pre-compact-save.js` の Bash全件フック

**pre-compact-save.js の5秒timeoutは「許容範囲」とは言えません。**  
理由は単純で、Bashは高頻度・連続発火しやすく、開発フローのホットパスだからです。  
timeout 5秒は「たまに重い」ではなく、**最悪ケースを大きくしすぎる**。  
許容できるのはせいぜい以下の条件を満たす場合のみです。

- 実測 p95 < 100ms
- p99 < 300ms
- timeout到達率 < 0.1%
- 失敗時に開発フローを止めない
- matcherが広すぎない
- Bash全件ではなく対象コマンド限定

この実測がない以上、**許容判断は不可能**です。

### 5. CodeGraph導入の影響
設計に問題あります。少なくとも以下の懸念があります。

1. **Write/Edit後の自動インデックスが粗い**
   - 変更量に関係なく動くなら無駄が大きい
   - 小変更1回ごとにインデックス更新はコスト過多

2. **排他制御の記述がない**
   - 連続Write/Editで多重起動したらどうするのか不明
   - SQLiteはロック競合の論点がある

3. **ROI計測hookがRead/Grep/Globごとに動作**
   - 計測のために観測コストを積む典型的な失敗設計
   - メトリクス取得が本体性能を侵食している可能性

4. **136MBバイナリ導入の重さ**
   - サイズ自体より、配布・更新・検証・キャッシュ戦略が未設計

5. **defer_loading=true だけでは安心材料にならない**
   - いつロードされるか
   - 初回レイテンシ
   - フェイル時挙動
   - キャッシュ無効化条件  
   が不明

CodeGraph自体は有用でも、**“便利だから hooks 3本で囲った”レベルの導入に見える**のが問題です。

### 6. セキュリティ
セキュリティ評価はかなり甘いです。

- **問題1: Goバイナリの供給網リスク**
  - 「公式リリース」だけでは不十分
  - **SHA256固定、署名検証、配布元真正性確認**が必要
  - 深刻度: **High**

- **問題2: hookの権限境界が曖昧**
  - 「node実行」は安全性の根拠ではない
  - NodeはファイルI/O・プロセス起動・ネットワークに触れます
  - 深刻度: **High**

- **問題3: shell injection対策が各hookで個別**
  - これは設計として悪い
  - 共通ライブラリ/共通ラッパーに寄せるべき
  - 深刻度: **High**

- **問題4: MCPサーバー数が多く攻撃面が広い**
  - 16+1サーバーで権限・接続先・データ流出経路の棚卸しが必要
  - 無効化済みが多いことは、逆に運用の複雑性を示す
  - 深刻度: **Medium**

- **問題5: セッション開始/終了hookの機微情報取り扱い不明**
  - handoff生成、injector、trace captureが何を保存するか不明
  - 深刻度: **Medium**

### 7. スケーラビリティ
「700+スキル、17 hook、16 MCP」で性能劣化は**十分あり得ます**。  
ただし、ここでも本数だけの議論は雑です。

スケーラビリティ上の本質的懸念は以下です。

- **探索空間の肥大化**
  - スキル707は発見可能性を下げる
  - ルーティング誤り・誤使用が増える

- **hookチェーンの予測不能性**
  - あるイベントで何本動くか把握しづらい
  - 遅延原因の切り分けが困難

- **MCP多重化による初期化/接続コスト**
  - deferしていても、依存グラフが複雑になる

- **設定の局所最適化**
  - グローバル設定とプロジェクト設定が分かれ、挙動が読みにくい

- **観測性不足**
  - p95/p99、タイムアウト率、競合率、キュー長などの指標がない

要するに、**今はまだ“規模が大きい”だけでなく、“規模を制御する仕組みがない”**のが問題です。
